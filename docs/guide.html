What is MCP?

The Model Context Protocol (MCP) is an open standard that lets AI assistants — such as Claude — discover and invoke external tools at runtime. Instead of being limited to the knowledge baked into the model, an MCP server exposes a set of tools that the assistant can call to read data, perform actions, or generate artifacts on behalf of the user.

Each tool has a name, a description, and a typed parameter schema. The AI reads these descriptions, decides which tool to call, and passes the correct arguments. The tool executes locally on your machine and returns a result that the AI incorporates into its response.

What is fabrica-persistence-mcp?

fabrica-persistence-mcp is an MCP server that gives Claude the ability to introspect relational database schemas, generate code using Liquid templates, produce dialect-specific DDL SQL, and create interactive ER diagrams. It supports SQL Server, PostgreSQL, MySQL, MariaDB, and SQLite.

It supports two complementary workflows:

  • Database-first — Introspect an existing database into db-design.json, then generate C# code via Liquid templates.
  • Design-first — Author db-design.json by hand to design a new schema, then generate DDL SQL and ER diagrams.

Typical things you can generate include:

  • Entity / model classes
  • Data-access repositories
  • Dialect-specific DDL SQL (CREATE TABLE, indexes, foreign keys, enums)
  • Interactive HTML ER diagrams
  • API DTOs or mapping code
  • Any text artifact you can express as a Liquid template

Installation

The application ships as a self-contained, single-file executable — no .NET runtime required. To install:

  1. Copy the executable to a permanent location on your machine.
  2. Either add that location to your PATH environment variable, or reference the executable by its full absolute path when registering it as an MCP server.

Registering in Claude Code

Option A — CLI

Run the following command in your terminal:

claude mcp add fabrica fabrica-persistence-mcp -- --project C:/projects/my-app

Replace C:/projects/my-app with the path to the directory that contains (or will contain) your persistence.project.json.

Option B — Manual JSON config

Add an entry to your Claude configuration file (~/.claude/claude_desktop_config.json): 

{
  "mcpServers": {
    "fabrica": {
      "command": "fabrica-persistence-mcp",
      "args": ["--project", "C:/projects/my-app"]
    }
  }
}

If the executable is not on your PATH, use the full path to the binary in the "command" field.

Getting Started

Once registered, open a Claude conversation and walk through these steps:

  1. 1
    Initialize the project — Ask Claude to run the init tool. This creates a persistence.project.json configuration file, JSON Schema files (persistence.project.schema.json and db-design.schema.json) for IDE autocompletion, starter Liquid templates, a .env file for credentials, and an AGENTS.md instruction file.
  2. 2
    Set the connection string — Open the .env file and replace the placeholder with your real database connection string. Keep this file out of version control.
  3. 3
    Introspect the schema — Ask Claude to run introspect_schema. This reads the connection from your config, introspects the database, and writes a db-design.json file containing the introspected metadata about your tables, columns, keys, and indexes. This file only contains what is actually in the database.
  4. 4
    Generate code — Ask Claude to run generate. Templates are rendered against the db-design.json data, merged with any explicit Relationships from your config, and output files are written to the configured OutputDir. No live database connection is needed for this step.
After any manual edits to persistence.project.json, ask Claude to run validate_config to catch errors before generating.
After upgrading fabrica-mcp, ask Claude to run update on existing projects. This refreshes AGENTS.md, persistence.project.schema.json, and db-design.schema.json so Claude knows about new features and IDE autocompletion stays current. All existing configuration is preserved.

Design-First Pipeline

In addition to the database-first workflow, fabrica-mcp supports a design-first pipeline where you author db-design.json by hand to design a new database schema, then generate DDL SQL and ER diagrams.

How It Works

  1. 1
    Design the schema — Write db-design.json with "Origin": "design". Define tables, columns, primary keys, foreign keys, indexes, and enums. The db-design.schema.json file provides IDE autocompletion.
  2. 2
    Generate DDL — Ask Claude to run generate_ddl with your target provider (e.g. postgresql). This produces a SQL file with CREATE TABLE, indexes, foreign keys, and enum types in the correct dialect.
  3. 3
    Visualize — Ask Claude to run generate_diagram to create an interactive HTML ER diagram. Open the file in any browser to review your schema visually with pan, zoom, and drag support.

The Origin Field

The Origin field in db-design.json controls overwrite protection:

ValueSet ByMeaning
"introspected"introspect_schema File was generated from a live database. introspect_schema can freely overwrite it.
"design"Hand-authored File was written for design-first use. introspect_schema refuses to overwrite and returns an error.

To switch a design-first schema back to database-first, delete db-design.json or change Origin to "introspected".

Design-First Schema Example

{
  "$schema": "db-design.schema.json",
  "Origin": "design",
  "Schemas": [
    {
      "Name": "public",
      "Tables": [
        {
          "Name": "Users",
          "Note": "Application users",
          "Columns": [
            { "Name": "Id", "DataType": "int", "IsPrimaryKey": true, "IsIdentity": true },
            { "Name": "Email", "DataType": "varchar(255)" },
            { "Name": "Role", "DataType": "UserRole" }
          ],
          "PrimaryKey": { "Columns": ["Id"] },
          "Indexes": [
            { "Columns": ["Email"], "IsUnique": true }
          ]
        }
      ]
    }
  ],
  "Enums": [
    {
      "Name": "UserRole",
      "Values": [
        { "Name": "Admin" },
        { "Name": "Editor" },
        { "Name": "Viewer" }
      ]
    }
  ]
}

ER Diagram Features

The generate_diagram tool produces a self-contained HTML file with:

  • Table boxes — column names, data types, and constraint icons (PK, FK, Unique, NOT NULL)
  • FK relationship lines — SVG lines connecting foreign key columns to their referenced tables
  • Enum boxes — distinct green-themed boxes listing enum values
  • Interactivity — pan (drag background), zoom (scroll wheel), drag tables (drag header), hover highlight on FK lines
  • FK-aware auto-layout — connected tables are placed adjacent to each other
Both generate_ddl and generate_diagram work with any db-design.json — whether hand-authored or introspected from a database. They also merge explicit Relationships from persistence.project.json if the config file exists.

Configuration Reference

The persistence.project.json file is the single source of truth for your project. It includes a $schema reference to persistence.project.schema.json, which provides IDE autocompletion and ensures that misspelled or unknown property names are caught immediately by validate_config. It contains the following top-level sections:

SectionRequiredPurpose
ConnectionYes Database connection with a Provider and ConnectionString.
OutputDirYes Directory where generated files are written (e.g. "src/Data").
TemplatesYes Named template definitions with Path, OutputPattern, Scope (PerModel | SingleFile), Mode (Always | SkipExisting), and optional AppliesTo (Tables | Views | All).
DefaultsNo Default values for Namespace, ContextName, Schema, IncludeViews, Include, and Exclude.
DataTypesNo Custom reusable type definitions (ClrType, optional MaxLength and DefaultValue).
TypeMappingsNo Maps database column types to CLR types or named DataTypes. Uses family-based inference: types like decimal(42,2) resolve via the base type decimal when no exact match exists.
RelationshipsNo Explicit foreign-key relationships that supplement introspected FKs. Merged at generation time (not written to db-design.json).
OverridesNo Per-class and per-property customizations: variant selection, DataType assignment, and Ignore flags.
LoggingNo Optional file-based logging (Enabled, LogPath, Level, RollingInterval, RetainedFileCountLimit).

Environment Variables

Database credentials should be kept out of version control. The .env file, located beside persistence.project.json, stores connection strings as simple KEY=VALUE pairs: 

# Database connection string — keep this file out of version control
DB_CONNECTION=Server=localhost;Database=MyDatabase;Trusted_Connection=true;

Reference variables in connection strings using ${VAR_NAME} syntax: 

"Connection": {
  "Provider": "sqlserver",
  "ConnectionString": "${DB_CONNECTION}"
}

System environment variables take precedence over values defined in .env. If a referenced variable is missing from both sources, the tool reports an error.

Tips

  • Validate after every edit — Always ask Claude to run validate_config after modifying persistence.project.json. It validates against the JSON Schema first (catching unknown or misspelled field names), then checks template files and semantic rules before you generate.
  • Use overrides instead of editing templates — For per-table or per-column tweaks (renaming types, ignoring columns, selecting variants) use the Overrides section rather than forking a template.
  • Start with SkipExisting stubs — Create a second template with "Mode": "SkipExisting" for hand-edited partial classes. These files are generated once and never overwritten.
  • Separate table and view templates — Use "AppliesTo": "Tables" or "AppliesTo": "Views" on template entries so entity templates only run for tables and view templates only run for views, avoiding empty output files.
  • Empty output is skipped — If a template renders to whitespace-only output (e.g. because of a guard condition), no file is written to disk and the file is reported as skipped.
  • Filter large schemas — Use Include / Exclude patterns in Defaults or pass models directly to generate to limit output to the tables you care about.
  • Review before committing — After generating, ask Claude to review the output files for correctness before committing them to source control.
  • Regenerate freely — Templates with "Mode": "Always" are designed to be overwritten. Keep manual changes in separate stub files.
  • Check AGENTS.md — The generated AGENTS.md file contains detailed instructions that Claude reads automatically. It covers the full template syntax, custom filters, macro/variant system, and override rules.
  • Run update after upgrading — When you install a newer version of fabrica-mcp, ask Claude to run update on each existing project. This overwrites AGENTS.md and JSON Schema files with the latest versions and normalizes your config file, all without losing any existing settings.
  • Always set Origin in design-first schemas — When hand-authoring db-design.json, set "Origin": "design" to prevent introspect_schema from accidentally overwriting your work.
  • Visualize before deploying — Run generate_diagram to review your schema visually before generating DDL. It's easier to spot missing relationships or naming issues in a diagram than in raw JSON.
  • Generate DDL for multiple dialects — You can run generate_ddl multiple times with different providers to produce SQL for each target database. Each run writes to a separate file (e.g. db-design.postgresql.sql, db-design.sqlite.sql).
